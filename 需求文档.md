# 《个人体系工坊》MVP 需求文档

> 暂定名：**个人体系工坊 / PersonaOS / MethodForge**
> 产品类型：AI Agent 产品 / 个人体系资产生成与维护平台
> MVP 目标：帮助用户把自己的经验、风格、偏好、规则和工作流，持续沉淀成 AI 可执行的个人体系资产。
> 核心一句话：**让 AI 不再随机生成，而是按照你的秩序工作。**

------

# 0. 先明确：这算不算 Agent 产品？

算。

但它不是那种“全自动帮你干完一切”的 Agent。

它更准确地说是：

> 一个围绕“个人体系沉淀”展开的 **Human-in-the-loop Agentic Workflow 产品**。

也就是说，它有 Agent 的特征：

| Agent 特征 | 在本产品中的体现                                  |
| ---------- | ------------------------------------------------- |
| 有目标     | 帮用户沉淀个人体系                                |
| 有记忆     | 保存用户风格、规则、偏好、历史版本                |
| 有推理     | 从素材中提炼原则、模式、禁忌                      |
| 有行动     | 生成 AGENTS.md、写作规则、Skill 草稿、知识库结构  |
| 有反馈     | 根据用户的“像 / 不像 / 太散 / 太空”等反馈修正体系 |
| 有迭代     | 每次喂养材料后持续更新个人规则库                  |
| 有约束     | 正式规则必须用户确认后才能写入                    |

但 MVP 阶段不建议做成复杂多 Agent 系统。

第一版应该是：

```txt
一个主 Agent
  +
几个稳定的工作流步骤
  +
一个可版本化的规则库
  +
人工确认机制
```

也就是：

> 先做“可控的 Agentic Workflow”，不要一上来做“不可控的 autonomous agent”。

------

# 1. 产品定位

## 1.1 产品名称暂定

可以先用一个中性名字：

# 个人体系工坊

英文可以叫：

# PersonaOS / MethodForge / SystemForge

我个人建议第一版中文品牌可以叫：

# 个人体系工坊

因为它比较容易让用户理解：

> 这是一个帮我打造自己做事体系的地方。

------

## 1.2 一句话定位

> 个人体系工坊是一个 AI 产品，帮助开发者、产品经理、写作者和知识工作者，把自己的经验、偏好、风格、规则和工作流，沉淀成可复用、可导出、可迭代的 AI 协作资产。

------

## 1.3 更有传播感的表达

> 不是让 AI 替你思考，而是让 AI 学会按照你的方式工作。

或者：

> 把你的经验，变成 AI 能听懂、能执行、能持续进化的个人系统。

------

# 2. 产品背景

AI 工具越来越强，但大量用户遇到的问题不是“AI 不会写”，而是：

- AI 每次都要重新解释背景
- AI 经常反复询问
- AI 输出风格不稳定
- AI 生成内容不像自己
- AI 写代码容易脱离项目规范
- AI 写文章容易变成通用模板腔
- AI 总结知识只是整理，不会真正沉淀
- 用户的经验散落在聊天记录、文章、代码、笔记、项目文档里，无法形成长期资产

本质问题是：

> 用户没有把自己的隐性经验，转化成显性的规则系统。

所以这个产品要解决的不是“生成更多内容”。

而是：

> 帮用户把自己过去的内容、反馈、经验、偏好，提炼成一套可复用的个人体系。

------

# 3. 核心问题定义

## 3.1 用户当前状态

很多用户现在是这样使用 AI 的：

```txt
每次开一个新对话
  ↓
重新解释背景
  ↓
AI 生成一版
  ↓
用户觉得不对
  ↓
继续补充要求
  ↓
AI 再改
  ↓
下次又重新来
```

这是一种低效循环。

它的问题是：

> 每次纠偏都没有沉淀成长期规则。

------

## 3.2 产品希望改变后的状态

使用本产品后，用户应该变成这样：

```txt
喂入材料
  ↓
AI 提炼个人偏好和规则
  ↓
用户确认
  ↓
规则进入个人体系库
  ↓
导出为 AGENTS.md / 写作风格 / Skill 草稿 / Prompt Pack
  ↓
真实任务中应用
  ↓
用户反馈
  ↓
体系继续进化
```

核心变化：

> 用户不是每次重新教 AI，而是让 AI 不断学习并维护一套长期属于自己的体系。

------

# 4. MVP 产品目标

MVP 不做大而全。

第一版只做一个最小但完整的闭环：

```txt
输入材料
  ↓
AI 提炼规则
  ↓
用户审核规则
  ↓
写入个人体系库
  ↓
导出可用资产
  ↓
在 Playground 里测试
  ↓
根据反馈继续修正
```

------

## 4.1 MVP 要验证的核心假设

| 假设                  | 说明                                       | 验证方式             |
| --------------------- | ------------------------------------------ | -------------------- |
| 用户愿意喂材料        | 用户愿意上传文章、代码、笔记、Prompt、反馈 | 看素材上传率         |
| AI 能提炼出有价值规则 | 从材料中识别用户风格、偏好、禁忌           | 看规则接受率         |
| 用户愿意审核规则      | 用户愿意点接受 / 修改 / 拒绝               | 看 proposal 处理率   |
| 导出的规则真的有用    | AGENTS.md / 写作指南能改善 AI 输出         | 看 Playground 满意度 |
| 体系能持续迭代        | 用户会多次回来喂新材料                     | 看 7 日 / 30 日留存  |

------

# 5. MVP 目标用户

MVP 不建议一开始面向所有人。

虽然最终愿景可以覆盖开发、产品、写作、知识管理，但第一版必须聚焦。

## 5.1 MVP 首批用户建议

优先做这三类：

| 用户类型                    | 为什么适合 MVP                                              |
| --------------------------- | ----------------------------------------------------------- |
| AI 高频开发者               | 他们已经感受到 AI 写代码跑偏，需要 AGENTS.md / Cursor Rules |
| 技术写作者 / 内容创作者     | 他们在意“AI 写出来像不像自己”                               |
| 独立开发者 / 个人品牌建设者 | 他们同时需要开发规则、写作风格、项目沉淀                    |

第一版最适合打的人群：

> **AI Native 独立开发者 / 技术写作者 / 想打造个人品牌的知识工作者。**

因为这类人既懂 AI 的价值，也最容易感受到“没有体系”的痛苦。

------

# 6. MVP 不做什么

这个很重要。

为了避免第一版失控，必须明确不做哪些东西。

| 暂不做                          | 原因                                   |
| ------------------------------- | -------------------------------------- |
| 不做复杂团队协作                | 第一版先验证个人体系沉淀               |
| 不做全自动修改规则              | 个人体系属于高价值资产，必须用户确认   |
| 不做真正多 Agent 自主行动       | 容易不可控，MVP 先做工作流 Agent       |
| 不做代码仓库深度分析            | 第一版可先支持文本/片段输入            |
| 不做完整 ChatGPT Skill 打包上传 | 可先生成 Skill 草稿，后续再支持 zip 包 |
| 不做浏览器插件                  | 先做 Web App 闭环                      |
| 不做复杂知识图谱                | 先用结构化规则库和版本记录             |
| 不做企业权限体系                | MVP 先个人版                           |

MVP 的原则：

> 先做“小而完整”，不要做“大而空”。

------

这是整个产品最关键的逻辑。

```txt
┌──────────────┐
│  1. 喂入素材  │
└──────┬───────┘
       ↓
┌──────────────┐

│ 接受/修改/拒绝 │
└──────┬───────┘
       ↓
┌──────────────┐
│  5. 写入体系库 │
│ Rulebase      │
└──────┬───────┘
```

# 8. MVP 核心功能模块

MVP 建议做 7 个模块。

------

## 8.1 Onboarding：用户画像初始化

### 目标

让系统先了解用户是谁、主要场景是什么、希望沉淀什么体系。

### 页面名称

```txt
/create-profile
```

### 用户需要填写

| 字段               | 示例                                    |
| ------------------ | --------------------------------------- |
| 你的身份           | 开发者 / 写作者 / 产品经理 / 独立开发者 |
| 主要使用场景       | AI 写代码、写文章、整理知识、产品文档   |
| 你希望 AI 记住什么 | 不要乱改架构、文章要有逻辑、不要营销腔  |
| 你讨厌 AI 什么行为 | 反复问、跑偏、生成空话、乱加依赖        |
| 你的输出偏好       | 信息密度高、有结构、有例子、有风格      |
| 你希望导出什么     | AGENTS.md / 写作风格 / Skill 草稿       |

### 系统输出

生成一个初始用户画像：

```md
# Personal Profile

## Identity
用户是一个偏系统型的独立开发者，同时关注写作和个人品牌。

## Core Preferences
- 喜欢结构清晰、信息密度高的内容
- 不喜欢 AI 输出空泛、散乱、模板化
- 希望 AI 在开发中遵守工程边界
- 希望将经验沉淀为可复用规则

## Current Focus
- 构建 AGENTS.md
- 沉淀写作风格
- 打造个人品牌体系
```

### MVP 要求

- 必须支持用户编辑画像
- 画像作为后续所有分析的基础上下文
- 画像不是一次性结果，后续可以被更新

------

## 8.2 Inbox：素材喂养箱

### 目标

让用户持续把材料喂给系统。

### 支持的素材类型

MVP 先支持文本，不做复杂文件解析。

| 类型            | 示例                             |
| --------------- | -------------------------------- |
| 文章            | 用户自己写过的博客、草稿         |
| 代码规则        | 现有 README、AGENTS.md、项目规范 |
| Prompt          | 过去用过的 Prompt                |
| AI 输出失败案例 | “这版不像我”“这版太散”           |
| 用户反馈        | “不是文字太多，而是分类不清”     |
| 项目描述        | 产品想法、开发规范、业务背景     |
| 知识笔记        | 复盘、想法、摘录                 |

### 素材录入方式

MVP 支持：

- 粘贴文本
- 手动选择素材类型
- 添加标签
- 添加一句说明

例如：

```txt
类型：AI 输出失败案例
说明：这篇博客写得太散，分点多但主线不强。
内容：……
```

### 素材状态

```txt
unprocessed    未分析
analyzed       已分析
used           已生成规则提案
archived       已归档
```

------

## 8.3 Analyzer：AI 分析器

### 目标

从素材中提炼出用户的：

- 偏好
- 禁忌
- 风格
- 重复模式
- 工作流
- 可复用规则
- 可导出资产建议

### 输入

```txt
用户画像
+
本次素材
+
已有规则库
```

### 输出

不是直接修改规则，而是生成候选洞察。

例如：

```json
{
  "insights": [
    {
      "type": "writing_preference",
      "title": "用户偏好高信息密度但结构清晰的长文",
      "evidence": "用户反馈：不是要减少文字，而是要分类更清楚",
      "suggested_rule": "长文可以保持足够篇幅，但必须通过分层标题、表格、流程图和示例降低阅读负担"
    },
    {
      "type": "ai_behavior_boundary",
      "title": "用户不希望 AI 误解反馈方向",
      "evidence": "用户指出：文字一大堆不是说文字太多",
      "suggested_rule": "当用户反馈内容太堆砌时，应优先优化结构与分类，而不是简单缩短内容"
    }
  ]
}
```

### 关键原则

Analyzer 不直接写入正式规则库。

它只能生成：

```txt
候选洞察
```

下一步必须进入 Proposal 审核。

------

## 8.4 Rule Proposal：规则变更提案

### 目标

让 AI 提出的体系更新变得可审核、可追踪、可拒绝。

### 为什么需要这个模块？

因为个人体系不能被 AI 自动污染。

如果 AI 每次分析完就直接改规则，时间长了用户体系会被模型“平均化”。

所以必须有一层：

```txt
AI 建议修改
  ↓
用户确认
  ↓
正式写入
```

### 提案结构

每条提案包含：

| 字段         | 说明                                    |
| ------------ | --------------------------------------- |
| 标题         | 这条规则解决什么问题                    |
| 类型         | 写作 / 开发 / 产品 / 知识管理 / AI 协作 |
| 当前观察     | AI 从素材中观察到了什么                 |
| 证据         | 来自哪段用户输入                        |
| 建议新增规则 | 具体要写入的规则                        |
| 影响资产     | 会影响 AGENTS.md / writing-style.md 等  |
| 操作         | 接受 / 修改 / 拒绝                      |

### 示例

```md
## 规则提案：长文不应简单压缩，而应优化结构

### 类型
writing_style

### 观察
用户并不是反对长文，而是反对没有分类、没有逻辑层级的文字堆砌。

### 证据
用户反馈：“文字一大堆的堆砌意思不是说文字太多，而是要加多并分点或者分类懂吗。”

### 建议写入规则
- 用户偏好高信息密度内容。
- 不要为了简洁而牺牲内容量。
- 当内容较长时，必须使用分类、表格、流程图、层级标题来提升可读性。
- 避免只写连续段落，应让读者能快速扫描结构。

### 影响文件
- writing-style.md
- personal-principles.md
```

用户可以：

```txt
接受
修改后接受
拒绝
稍后处理
```

------

## 8.5 Rulebase：个人体系规则库

### 目标

保存用户已经确认的正式规则。

这是产品的核心资产。

### Rulebase 的结构

MVP 可以分成 6 类：

```txt
personal-profile.md
个人画像

principles.md
通用原则

ai-collaboration.md
AI 协作规则

coding-style.md
开发规则

writing-style.md
写作规则

knowledge-system.md
知识沉淀规则
```

### 示例结构

```txt
rulebase/
  personal-profile.md
  principles.md
  ai-collaboration.md
  coding-style.md
  writing-style.md
  knowledge-system.md
  changelog.md
```

------

## 8.6 Asset Builder：资产生成器

### 目标

把规则库转换成具体可用的资产。

MVP 先支持 4 种导出。

| 导出类型               | 作用                                     |
| ---------------------- | ---------------------------------------- |
| `AGENTS.md`            | 给 AI coding agent 使用的项目规则        |
| `writing-style.md`     | 写作风格指南                             |
| `personal-system.md`   | 个人体系总览                             |
| `skill-draft/SKILL.md` | ChatGPT Skill 草稿，不一定第一版支持打包 |

### MVP 导出优先级

第一优先级：

```txt
AGENTS.md
writing-style.md
personal-system.md
```

第二优先级：

```txt
Skill Draft
Cursor Rules
Claude Instructions
```

------

### 8.6.1 AGENTS.md 导出内容

MVP 生成的 `AGENTS.md` 应该包括：

```md
# AGENTS.md

## Project / User Context

## Core Principles

## AI Collaboration Rules

## Coding Style

## Project Structure Preferences

## Boundaries

## Validation Requirements

## Communication Style
```

示例片段：

```md
## AI Collaboration Rules

- Before editing code, restate the task and identify likely affected files.
- Do not rewrite large areas of the project unless explicitly requested.
- Do not introduce new dependencies without explaining why existing tools are insufficient.
- Prefer small, safe, reviewable changes.
- If requirements are ambiguous, make a reasonable assumption and state it clearly instead of repeatedly asking.
```

------

### 8.6.2 writing-style.md 导出内容

```md
# Writing Style Guide

## Core Voice

## Structure Preferences

## What to Avoid

## Preferred Blog Structure

## Examples of Good Output

## Revision Rules
```

示例片段：

```md
## Structure Preferences

- Prefer dense, idea-driven long-form writing.
- Do not reduce content only for brevity.
- Use sections, tables, diagrams, examples, and progressive logic to improve readability.
- Each section should advance the central argument instead of becoming isolated bullet points.
```

------

### 8.6.3 personal-system.md 导出内容

```md
# Personal System

## Identity

## Current Focus

## Core Beliefs

## Working Style

## AI Usage Philosophy

## Development System

## Writing System

## Knowledge System

## Brand Direction
```

这个文件适合用户自己看，也适合作为个人品牌方法论的入口。

------

## 8.7 Playground：体系测试场

### 目标

让用户用自己的规则测试 AI 输出是否更像自己。

### 典型场景

用户选择一个体系资产，例如：

```txt
writing-style.md
```

然后输入任务：

```txt
帮我写一篇关于 AGENTS.md 的认知篇博客。
```

系统调用：

```txt
用户画像
+
writing-style.md
+
任务
```

生成结果。

然后用户反馈：

```txt
像我
不像我
太散
太空
太短
太模板
逻辑不够
例子不够
```

系统把反馈转化为新的规则提案。

------

# 9. Agent 工作流设计

MVP 不做复杂多 Agent，但可以内部拆成几个逻辑角色。

## 9.1 Agent 总体架构

```txt
User
 ↓
Orchestrator Agent
 ↓
┌──────────────────────┐
│ Profile Reader        │ 读取用户画像
│ Material Analyzer     │ 分析素材
│ Rule Extractor        │ 提炼规则
│ Proposal Generator    │ 生成变更提案
│ Asset Builder         │ 导出资产
│ Playground Evaluator  │ 根据反馈修正体系
└──────────────────────┘
 ↓
Rulebase / Artifacts / Changelog
```

这些不一定要做成真正独立 Agent。

MVP 可以用一个后端服务，根据不同任务调用不同 Prompt Chain。

------

## 9.2 Agent 的核心任务

### 任务 1：分析素材

输入：

```txt
用户画像
已有规则
新素材
```

输出：

```txt
洞察列表
候选规则
证据
建议影响范围
```

------

### 任务 2：生成规则提案

输入：

```txt
洞察
已有规则
```

输出：

```txt
新增规则
修改规则
删除规则
原因
影响文件
```

------

### 任务 3：应用规则变更

输入：


# 10. Agent 行为原则

这是产品内部 Agent 必须遵守的系统规则。

## 10.1 不能自动污染用户体系

```txt
AI 可以提出规则变更。
AI 不能直接修改正式规则。
正式规则必须用户接受后才能写入。
```

------

## 10.2 每条规则必须有来源

每条规则最好带证据：

```txt
这条规则来自哪段素材？
来自哪次用户反馈？
来自哪次输出失败？
```

这样用户才会信任系统。

------

## 10.3 不确定时生成候选，不要武断归纳

比如用户只说了一次：

```txt
这次写得太短了。
```

AI 不能直接总结：

```txt
用户永远喜欢长文。
```

更合理的提案是：

```txt
候选观察：在认知类博客场景中，用户偏好内容更充分，而不是过度简短。
建议规则：当用户要求博客/认知文章时，优先保证内容密度和结构完整。
```

------

## 10.4 规则要可执行

不接受这种规则：

```txt
写得好一点。
代码优雅一点。
表达高级一点。
```

要转化为：

```txt
每篇文章必须有明确主论点。
长文必须使用层级标题和辅助表格。
开发任务中不要修改无关文件。
新增依赖必须说明理由。
```

```txt
“长文应保持信息密度，但必须使用层级结构和辅助图表。”

系统写入：
writing-style.md

用户导出：
writing-style.md

用户测试：
让 AI 写一篇新博客

用户反馈：
这次更像我了，但例子还不够。

系统继续生成：
“每个核心观点至少配一个具体例子”的规则提案。
```

# 21. 产品风险与解决方案

## 风险 1：用户不知道该喂什么

### 解决方案

提供素材模板：

```txt
粘贴一篇你写过的文章
粘贴一次你不满意的 AI 输出
粘贴你的项目 README
粘贴你常用的 Prompt
写下你最讨厌 AI 犯的 3 个错误
```

------

## 风险 2：AI 提炼的规则太空

### 解决方案

规则必须满足三个条件：

```txt
具体
可执行
有证据
```

不允许生成：

```txt
写得更好
保持专业
代码优雅
```

必须转成：

```txt
每篇文章必须有一个主论点。
不要连续输出超过 5 段无标题长文。
新增依赖必须说明原因。
```

------

## 风险 3：规则越积越多，变成垃圾堆

### 解决方案

增加规则状态：

```txt
active
archived
candidate
```

并且后续支持：

```txt
相似规则合并
过期规则提醒
规则冲突检测
```

MVP 可以先人工管理。

------

## 风险 4：用户不信任 AI 自动修改

### 解决方案

坚持：

```txt
AI 只提案，不自动写入正式规则。
```

这是产品信任基础。

------

## 风险 5：产品概念太抽象

### 解决方案

MVP 首页不要讲太多大词。

直接讲场景：

```txt
让 AI 写文章更像你
让 AI 写代码更守规矩
把你的经验沉淀成 AGENTS.md、写作风格和个人体系文档
```

# 24. MVP 的核心差异化

这个产品不能被做成普通 AI 文档生成器。

普通 AI 工具是：

```txt
输入需求
  ↓
生成内容
  ↓
结束
```

你的产品必须是：

```txt
输入材料
  ↓
提炼用户体系
  ↓
用户确认
  ↓
形成规则资产
  ↓
应用到真实任务
  ↓
反馈继续进化
```

核心差异是：

> 别人帮用户生成内容，你帮用户生成“生成内容的方式”。

这就是产品壁垒。

# 25. 一句话产品愿景

> 每个人未来都不只需要一个简历，而需要一套可执行的个人系统。
> 这个系统记录他的原则、审美、经验、工作流和判断方式，让 AI 不再随机生成，而是按照他的秩序工作。

------

# 26. 最终 MVP 定义

这个产品的 MVP 可以这样定义：

> 一个 Web App，允许用户持续喂入文章、代码规范、Prompt、反馈和项目材料，由 AI 提炼出个人偏好与工作规则，经过用户确认后写入个人体系库，并导出为 AGENTS.md、写作风格指南和个人体系文档，从而让用户在开发、写作和知识沉淀中获得更稳定、更像自己的 AI 输出。

# 27. 开发优先级总结

最重要的是这条主线：

```txt
Material
  ↓
Insight
  ↓
RuleProposal
  ↓
Rulebase
  ↓
Artifact
```

开发时不要偏离这条线。

第一版只要这条链路跑通，这个产品就成立。

页面可以简陋。
UI 可以先简单。
导出类型可以少。
但这个闭环不能断。

因为这个项目真正的灵魂不是界面，而是：

> 把人的隐性经验，转化成 AI 可执行规则的能力。
